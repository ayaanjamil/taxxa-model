## Problem Statement

Finnish accountants ask multi-hop regulatory questions — e.g., "what withholding-tax
rate applies to a foreign specialist with key-personnel status, and how long is the
tax card valid?" The answer spans multiple documents: a statute (Finlex), a Tax
Administration bulletin (Verohallinto), and an amendment that changed both the rate
and the validity window.

Naive RAG fails because:
- Fixed chunks split legal clauses mid-rule
- Vector similarity is recency/authority-blind (repealed 2014 law ≈ 2024 amendment)
- Cross-references ("See §102 AVL") are plain text to the index — not followed
- Multi-hop questions get one-hop answers

Corpus: Finlex (Finnish statutes, full amendment history, court rulings KHO/KKO)
and Verohallinto (tax authority guidance, bulletins, interpretations).
83 graded QA pairs for eval. Judged on: correctness > groundedness > approach.

## What Makes This Cool

- A visual graph where judges can literally watch the system traverse nodes to reach an answer
- Eval-driven development: ship a working baseline at hour 10, then every graph feature
  you add is justified by real eval failures on the 83 QA pairs
- Always-working system: risk is bounded — you can demo at any point

## Constraints

- Solo developer, Python, OpenRouter API
- 48 hours total
- New to both graph databases and RAG pipelines (building with AI assistance)
- No Neo4j — in-memory networkx to keep debugging simple
- Must have eval running early (measure before you optimize)

## Premises

1. networkx + JSON beats Neo4j for a 48-hour solo build on this corpus size — agreed
2. LLM-assisted relation extraction beats hand-written Finnish legal parsers — agreed
3. Getting eval working in hour 2-3 is the highest-value early investment — agreed
4. Graph visualization frontend is a judging differentiator — agreed

## Approaches Considered

### Approach A: Structured RAG with Citation Resolver
Parse corpus preserving hierarchy (section/paragraph), embed each unit, vector
search + regex-based cross-ref resolver, LLM answers with §IDs.
- Effort: M (~8-10 hrs human / ~2 hrs CC scaffolding)
- Risk: Low
- Ceiling: Medium eval score; multi-hop tier 4-5 questions will miss

### Approach B: Full GraphRAG from scratch
Build typed graph first (amends/repeals/interprets/references edges via LLM extraction),
then hybrid retrieval, then 4-agent loop. Beautiful architecture, high risk.
- Effort: XL (~20-24 hrs human / ~5 hrs CC)
- Risk: High (complex, may not finish)
- Ceiling: High eval score if it works

### Approach C: Layered Build — A then B (CHOSEN)
Ship Approach A first (working baseline by hour 10). Run eval on 83 QA pairs.
Find failure clusters. Layer graph edges exactly where failures occur.
Builds to look like B but always has a working system.
- Effort: L (~16-18 hrs human / ~4 hrs CC)
- Risk: Low-Med (baseline always demoable)
- Ceiling: High — eval guides which graph features matter most

### Phase 2: Structured RAG Baseline (Hours 3–10)

**Parser (structure-aware, not chunk-based):**
```python
# Target node shape
{
  "id": "AVL_§102_1",           # statute_section_paragraph
  "source": "finlex",
  "statute": "AVL",
  "section": "102",
  "paragraph": "1",
  "text": "...",
  "date": "2024-01-01",         # effective date
  "superseded_by": None,        # or "AVL_§102_1_amendment_2024"
  "references": ["AVL_§33", "TVL_§77"],  # extracted §refs
  "embedding": [...]            # computed after
}
```

Parse Finlex HTML/XML preserving statute → section → paragraph hierarchy.
Parse Vero documents preserving document → section structure.
Extract explicit §references with regex: `§\s*(\d+[a-z]?)\s*(momentti\s*\d+)?`
Store all nodes as JSON (nodes.json).

**Embeddings:**
Use `sentence-transformers/paraphrase-multilingual-mpnet-base-v2` locally (handles Finnish
morphological inflection — "verovuosi" vs. "verovuotta" etc.). Do NOT use `text-embedding-3-small`
— it is English-optimized and will degrade cosine similarity on Finnish text significantly.
Alternatively: OpenRouter `multilingual-e5-large` if local model setup is slow.
Store as numpy array + id→index map (vectors.npy, id_map.json). ~1000-3000 nodes expected.

**Retrieval:**
```python
def retrieve(query, top_k=5):
    query_vec = embed(query)
    scores = cosine_similarity(query_vec, all_vectors)
    top_ids = top_k by score
    
    # Expand: follow §references mentioned in retrieved nodes
    expanded = set(top_ids)
    for node_id in top_ids:
        expanded.update(nodes[node_id]["references"])
    
    return [nodes[id] for id in expanded]
```

### Phase 3: Graph Layer — Typed Edges (Hours 10–18)

After seeing eval failures, add graph structure where it helps.

**Graph schema (networkx DiGraph):**
```python
# Node types
STATUTE     # AVL, TVL, EPL — the full act
ARTICLE     # §102 AVL — individual section  
CLAUSE      # paragraph within an article
GUIDANCE    # Verohallinto bulletin (document-level)
GUIDANCE_S  # section of a Vero bulletin
COURT_CASE  # KHO/KKO ruling

# Edge types (typed as edge attribute)
"amends"        # newer version → older (critical for recency)
"repeals"       # replaces entirely
"references"    # explicit §cross-reference
"interpreted_by" # Statute → Vero section that interprets it
"cites"         # Vero doc → Statute section
"overrides"     # Court ruling → Statute section
```

**LLM-assisted relation extraction:**
```python
# For each node pair where one references the other by text:
prompt = """Extract the relationship between these two legal documents.
Doc A: {doc_a_text}
Doc B: {doc_b_text}
Return JSON: {"relation": "amends|repeals|references|interpreted_by|cites|overrides",
              "confidence": 0.0-1.0,
              "reasoning": "..."}"""
```

Run extraction on pairs where a reference is already detected (regex found §ID).
Filter: only add edge if confidence ≥ 0.7. On malformed JSON or unknown relation type,
fall back to edge type "references" with confidence 0.5 (don't crash).
Store as networkx edges with relation type + confidence.

**Time-box rule for Phase 3:** If LLM extraction is not producing valid edges by hour 14,
fall back to using only the `references` field from Phase 2 node parsing (regex-detected §IDs)
as edges. This still gives you graph traversal without the extraction overhead.

**Upgraded retrieval with graph walk:**
```python
def retrieve_with_graph(query, top_k=5):
    entry_points = vector_retrieve(query, top_k)
    
    neighborhood = set(entry_points)
    for node_id in entry_points:
        # Walk typed edges (correct networkx API: neighbors() returns node IDs only)
        for neighbor in G.successors(node_id):
            data = G[node_id][neighbor]
            if data["relation"] in ["interpreted_by", "cites", "amends"]:
                neighborhood.add(neighbor)
        # Check supersession: if this node is superseded, include the newer version
        for pred in G.predecessors(node_id):
            if G[pred][node_id]["relation"] == "amends":
                neighborhood.add(pred)  # add the newer amendment
    
    # Rank: Finlex statute > Verohallinto guidance > KHO court ruling
    # (statute is primary law; guidance interprets it; court rulings clarify edge cases)
    # Within same type: prefer non-superseded, then most recent date
    TYPE_PRIORITY = {"ARTICLE": 0, "CLAUSE": 0, "GUIDANCE_S": 1, "GUIDANCE": 1, "COURT_CASE": 2}
    def node_sort_key(nid):
        n = nodes[nid]
        superseded = 1 if n.get("superseded_by") else 0
        type_rank = TYPE_PRIORITY.get(n["type"], 3)
        date_rank = -(n.get("date") or "0000")  # newer = lower rank value
        return (superseded, type_rank, date_rank)
    return sorted(neighborhood, key=node_sort_key)
```

Run eval after graph layer. Expected improvement: +10-20% on hard questions.

### Phase 4: Agent Loop (Hours 18–24, if time allows)

Add a planner for multi-hop decomposition:

```python
PLANNER_PROMPT = """Decompose this Finnish tax law question into 2-4 sub-questions
that can each be answered by looking up a specific statute or guidance document.

Question: {question}
Return JSON: {"sub_questions": ["...", "...", "..."]}"""

def agent_answer(question):
    # Step 1: Planner decomposes
    sub_questions = planner(question)
    
    # Step 2: Retrieve for each sub-question
    all_evidence = []
    traversal_log = []
    for sq in sub_questions:
        nodes, hops = retrieve_with_graph(sq)  # returns (nodes, traversal hops)
        all_evidence.extend(nodes)
        traversal_log.extend(hops)
    
    # Dedupe by node ID, keep highest retrieval score
    seen = {}
    for n in all_evidence:
        if n["id"] not in seen or n["score"] > seen[n["id"]]["score"]:
            seen[n["id"]] = n
    deduped_evidence = list(seen.values())
    
    # Step 3: Draft answer
    draft = generate_answer(question, deduped_evidence)
    
    # Step 4: Verifier — check each claim has a source node
    verified = verifier(draft, all_evidence)
    
    return verified, traversal_log
```

Verifier prompt (make optional via `ENABLE_VERIFIER=true` env var — adds ~3s latency per answer):
```
For each claim in this answer, check that it is supported by a specific provided
document section. Mark unsupported claims. Do not include claims without sources.
The fact and answer may be in Finnish. Evaluate semantically, not lexically.
```

### Phase 5: Frontend (Hours 24–36, after baseline eval is working)

**Stack:** Next.js (App Router) + Cytoscape.js
**Backend:** Python FastAPI (localhost:8000) — all graph/RAG/eval logic stays in Python
**Frontend:** Next.js (localhost:3000) — proxies to FastAPI via API routes or direct fetch
**Scope:** Static graph render after answer returns. NO real-time animation — that's
8+ hours for a solo developer and judges don't need it. A static Cytoscape.js subgraph
render is 2-3 hours and provides the same visual differentiator.

Next.js setup (fastest path):
```bash
npx create-next-app@latest frontend --typescript --tailwind --app
cd frontend && npm install cytoscape @types/cytoscape
```

**FastAPI backend:**
```python
@app.post("/ask")
async def ask(q: QuestionRequest):
    answer, traversal_log = agent_answer(q.question)
    # traversal_log: list of {"from": node_id, "to": node_id, "relation": rel_type}
    # Collect unique nodes referenced in the log
    node_ids = set(h["from"] for h in traversal_log) | set(h["to"] for h in traversal_log)
    return {
        "answer": answer,
        "graph": {
            "nodes": [{"id": nid, "type": nodes[nid]["type"],
                       "label": nodes[nid]["id"],
                       "superseded": bool(nodes[nid].get("superseded_by"))}
                      for nid in node_ids if nid in nodes],
            "edges": [{"source": h["from"], "target": h["to"], "type": h["relation"]}
                      for h in traversal_log]
        }
    }
```
